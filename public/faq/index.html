If you do not find answers to your questions here, please ask them in our mailing lists (AppDF mailing list, OpenIAB mailing list), or just tweet us.

AppDF - Application Description File

What is the current project status?

  • The current version of AppDF specification is 1.0.
  • Two appstores - Opera Mobile Store and Yandex.Store - have AppDF publishing support in the beta stage.
  • Several more appstores have committed to supporting AppDF and are now working on implementation.
  • AppDF Editor is ready.
  • A command line tool for submitting AppDF to appstores is in beta.

Where can I submit AppDF files?

You can currently submit them to Opera Mobile Store and Yandex.Store. Both have AppDF format support on their developer portals. Several other appstores are working on AppDF implementation.

How can I create AppDF file?

You can either follow AppDF specification or use AppDF Editor.

Can I create one AppDF for all stores, including Google Play?

Yes, you can. When we created AppDF specification, we carefully checked all the existing Android appstores to make sure AppDF format includes all the necessary information. AppDF also supports store-specific sections, so you can add custom information for individual appstores.

What is the difference between AppDF and mobile app aggregators?

AppDF is a format that lets developers upload single distribution files directly to multiple appstores. In other words, the AppDF approach cuts out the middle man, enabling app developers to publish their apps directly to appstores rather than to an aggregator.

Which Android appstores support the AppDF format?

CodeNGO, Opera Mobile Store, SlideME and Yandex.Store already support submissions using AppDF. Appland and Aptoide are working on integrating support.

How can I describe In-App Purchases in AppDF?

Right now, AppDF does not support in-app purchases. This is because it is currently very difficult to create one APK file to support in-app billing API for different appstores. We plan to support in-app purchases in the next version of AppDF. It will be aligned with the OpenIAB library, facilitating the creation of an APK file that supports in-app billing for all the existing appstores.

What should I do if I have a proposal for the next version of AppDF specification?

The best thing to do is join the mailing list, and describe the motivation (e.g. use cases) and optionally the proposal itself. You can also submit a pull request on GitHub.

OpenIAB - Open In-App Billing

What is current project status?

OpenIAB library is currently under development. We have finished a sample game that supports in-app billing for four different stores (Google Play, Amazon AppStore, Samsung Apps and SK T-Store) and are now working on splitting in-app purchase code in a separate library. We will try to keep its interface as close to Google Play IAB API as possible. We have simultaneously starting work on OpenIAB API that other stores could implement to support all the apps and games that use OpenIAB Lib. Join us to develop OpenIAB together.

Does OpenIAB support mobile operator billing?

This question is based on a misunderstanding. All payments are processed directly by each appstore. There is no middle man standing between app developers and appstores. OpenIAB will not process payments for the appstores. It is simply an API enabling apps to communicate with appstores to request in-app billing and get a response.

What is the difference between OpenIAB Lib and OpenIAB API?

OpenIAB Lib is a library that wraps in-app billing APIs of several appstores. All you need to do is integrate it once, and your app will support in-app billing for all these stores. OpenIAB API is yet another in-app billing API that appstores can implement. OpenIAB API is integrated into OpenIAB Lib as one of the supported APIs. By implementing OpenIAB API, the appstores will automatically support any app that uses OpenIAB Lib.

General OPF Questions

Who is standing behind OPF?

Yandex and Opera have provided resources to make it happen, and contributors to OnePF have provided actual work. Not to forget the stores that have implemented or a working on support for AppDF.

How are decisions about the next version of the standards made

So far we haven't got a formal charter, but we are working on one. Our goal is that "rough consensus and running code" are the main guidance.
